# Zam 1.3 CSV Export Foundation

## Summary

Zam 1.3 should add a reliable CSV export flow that matches the current import contract without expanding the importer. The goal is simple data portability: users can export their transaction data with a small set of practical filters, then import that same file back into Zam without data shape surprises.

This is a core data-portability release, not a reconciliation or accountant-export release.

## Goal

Allow users to reliably export transactions with minimal configuration.

The v1.3 success test:

```text
Export CSV
Delete or reset local data
Import CSV
Confirm the same transaction data comes back cleanly
```

## Scope

### Export CSV

Add a clear export entry point that generates a CSV file from the user's transaction data.

The export should:

- Use the user's currently available/decrypted app data.
- Not require Cloud Sync.
- Not upload transaction data to any server just to export.
- Respect browser-only and signed-in users the same way.

### Custom File Name

Let the user choose or confirm the downloaded file name.

Examples:

- `June Budget.csv`
- `Checking Account.csv`
- `Zam Export 2026-06-18.csv`

Default recommendation:

```text
zam-transactions-YYYY-MM-DD.csv
```

File names must:

- End in `.csv`.
- Remove unsafe filename characters.
- Fall back to the default name if the user clears the field.

### Date Range

Support a small set of practical date ranges:

- Current Month
- Last Month
- Custom Range
- All Time

Dates must be interpreted using Zam transaction dates, not browser export time. Export dates should use each transaction's stored date only. Do not shift dates based on browser timezone or export time.

### Account Filter

Allow filtering by payment method/account.

Examples:

- All Accounts
- Checking
- Savings
- Apple Card

Use existing transaction payment-method data. Do not introduce a new account schema in v1.3.

### Category Filter

Allow filtering by category.

Examples:

- All Categories
- Groceries
- Dining
- Income

Use existing category/transaction data. Do not introduce new category logic in v1.3.

### Stable Date Format

Every exported row must use:

```text
YYYY-MM-DD
```

Example:

```text
2026-06-01
```

Do not export locale-dependent date strings such as:

- `6/1/2026`
- `June 1, 2026`
- `01/06/2026`

This is a hard requirement because Zam imports should not depend on browser locale.

### Default Columns Only

Export these columns in this order:

```text
Date
Description
Amount
Category
Account
Notes
```

The exported CSV must include this header row:

```text
Date,Description,Amount,Category,Account,Notes
```

Column meanings:

- `Date`: transaction date in `YYYY-MM-DD`.
- `Description`: transaction description.
- `Amount`: signed decimal amount.
- `Category`: transaction category.
- `Account`: current payment method/account value.
- `Notes`: transaction notes.

## Amount Handling

Use one `Amount` column in v1.3 exports.

Recommended convention:

- Expenses export as negative values.
- Income/deposits export as positive values.
- Savings/debt records should follow the current transaction type semantics already used by the app.

Examples:

```text
-42.15
2600.00
```

Amounts should export as plain decimals with no currency symbols or thousands separators.

Good:

```text
-42.15
2600.00
```

Avoid:

```text
$42.15
1,234.56
($42.15)
```

Do not add separate `Debit` and `Credit` export columns in v1.3. The importer can handle debit/credit files from banks, but Zam's own export should be simpler and stable.

## Out of Scope

Do not add these in v1.3:

- Custom column mapping.
- Custom column ordering.
- Saved export templates.
- Excel-specific formatting.
- Accountant-specific reports.
- Google Drive export.
- Dropbox export.
- Import reconciliation.
- Transfer detection.
- Merchant cleanup.
- Category intelligence.
- Statement balancing.

## UX Requirements

The export UI should answer four questions:

```text
What date range am I exporting?
Which accounts are included?
Which categories are included?
What file name will download?
```

Recommended default state:

```text
Date range: Current Month
Account: All Accounts
Category: All Categories
File name: zam-transactions-YYYY-MM-DD.csv
```

Show the export count before download:

```text
68 transactions will be exported
```

Primary action:

```text
Export CSV
```

If filters produce zero rows, disable export or show a clear empty-state message:

```text
No transactions match these filters.
```

## Import Compatibility Requirement

Every Zam export should be a valid Zam import.

Acceptance criteria:

- Exported `Date` imports as the same date.
- Exported `Description` imports as the same description.
- Exported `Amount` imports as the same amount and type.
- Exported `Category` imports as the same category text.
- Exported `Account` imports as the same payment method/account text.
- Exported `Notes` imports as the same notes text.
- CSV escaping handles commas, quotes, and newlines safely.

## Privacy and Legal Notes

Expected legal impact:

- Privacy/UX copy should make clear that exported CSV files are plain CSV files after download. They are not encrypted by Zam after they leave the app, so users should store them carefully.
- No additional Privacy Policy change should be required if export stays fully local and does not transmit data.
- No Terms change should be required unless export changes data-loss, recovery, billing, or support promises.
- Changelog should be updated because CSV export is user-visible.

Before release, run the major-change gate:

```text
documentation/Major-Change-Release-Gate.md
```

## Test Plan

Minimum tests:

- Export all transactions.
- Export current month.
- Export last month.
- Export custom date range.
- Export all accounts.
- Export one account/payment method.
- Export all categories.
- Export one category.
- Verify `YYYY-MM-DD` date format.
- Verify exported dates do not shift due to browser timezone or export time.
- Verify the header row is exactly `Date,Description,Amount,Category,Account,Notes`.
- Verify amounts are plain decimals with no currency symbols or thousands separators.
- Verify filenames end in `.csv` and unsafe filename characters are removed.
- Verify export count updates before download.
- Verify CSV escaping for commas, quotes, and newlines.
- Verify empty filter state.
- Import a Zam export back into Zam and confirm the data shape survives.

## Release Label

Target release:

```text
Zam 1.3 - CSV Export Foundation
```

Keep v1.3 focused. If the feature starts becoming reconciliation, reporting, or cloud export work, move that to a later release.
