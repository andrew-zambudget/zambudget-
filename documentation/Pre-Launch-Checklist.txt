# Zam! Pre-Launch Checklist

Status: living checklist
Priority: high

Use this checklist before a broad public launch, provider cutover, or paid-user push. It is not a feature backlog.

## 1. Privacy-First Visibility

- [ ] Decide whether visitor analytics are enabled.
- [ ] If analytics are enabled, use privacy-conscious aggregate analytics only.
- [ ] Avoid ad trackers, data brokerage, fingerprinting, or cross-site profiling.
- [ ] Update `zambudget.com/privacy`, `zambudget.com/cookie-policy`, and `zambudget.com/trust` before enabling or changing telemetry.
- [ ] Confirm app pages remain `noindex,follow`.

Current public stance: the marketing website may use Cloudflare aggregate delivery, security, performance, and web-analytics telemetry. The app should not add third-party ad tracking or fingerprinting.

## 2. Error Monitoring

- [ ] Decide whether production error monitoring is needed.
- [ ] If enabled, document what metadata is collected and where it is sent.
- [ ] Do not send budget contents, recovery keys, tokens, full card numbers, gift card numbers, PINs, or private exports to monitoring tools.
- [ ] Update legal/trust docs before enabling a third-party monitoring provider.

## 3. Feedback Loop

- [ ] Confirm website feedback routes to `support@zambudget.com`.
- [ ] Confirm support docs tell users not to send recovery keys, magic links, passwords, full card numbers, gift card numbers, PINs, or private budget exports.
- [ ] Confirm public Contact Us, Help Center, Guides, and Trust Stack pages are current.

## 4. Performance And Accessibility

- [ ] Run Lighthouse or an equivalent browser audit on key marketing pages.
- [ ] Run app smoke tests on desktop and mobile viewports.
- [ ] Confirm focus states, keyboard navigation, visible labels, and reduced-motion behavior on major flows.
- [ ] Confirm screenshots/images are optimized and current.

## 5. Legal And Trust

- [ ] Complete `documentation/Major-Change-Release-Gate.md`.
- [ ] Review app `privacypolicy.html` and `terms.html`.
- [ ] Review marketing `privacy.html`, `terms.html`, `cookie-policy.html`, and `trust.html`.
- [ ] Review the public changelog and roadmap.
- [ ] Confirm Cloud Sync copy accurately says synced budget contents are encrypted client-side and require the user's recovery key or trusted local browser key.
- [ ] Confirm browser-only local budgets are documented as encrypted at rest but unrecoverable if local browser storage or local vault key material is lost.
- [ ] Confirm local browser encryption copy does not claim protection while the app is open and decrypted in memory.

## 6. Billing

- [ ] Do not change Stripe behavior unless billing work is in scope.
- [ ] Confirm `config.json` billing mode matches the intended launch state.
- [ ] Confirm Stripe Checkout, Billing Portal, webhooks, cancellation, refund, and account-deletion behavior before accepting paid users.
- [ ] Confirm public pricing and Premium copy match actual product behavior.

## 7. Deployment Recovery

- [ ] Read `START_HERE.md`.
- [ ] Read `documentation/App-Rebuild-Runbook.md`.
- [ ] Read `documentation/Infrastructure-Inventory.md`.
- [ ] Confirm GitLab, Cloudflare, Supabase, Stripe, Proton Mail, and registrar owner recovery paths.
- [ ] Confirm secrets can be restored from a password manager or provider dashboard.
- [ ] Confirm no secret values are committed.

## Launch Rule

If the release changes auth, Cloud Sync, billing, account deletion, data retention, telemetry, public promises, SEO posture, or provider behavior, update legal/docs/changelog before calling it done.
