Last updated: 2026-06-18

Use this checklist whenever app auth, Supabase Auth, SMTP, Proton Mail, OAuth, or app-domain routing changes.

## Required Production Settings

Supabase Auth URL configuration:

- Site URL: `https://app.zambudget.com`
- Redirect URLs must allow the production app:
  - `https://app.zambudget.com`
  - `https://app.zambudget.com/`
  - `https://app.zambudget.com/index.html`
  - `https://app.zambudget.com/login.html`
  - `https://app.zambudget.com/**` if using wildcard redirect rules
- Remove old-brand URLs such as `https://app.budget-buddy.io/**` unless a deliberate temporary migration is active.
- Do not leave `localhost` or `127.0.0.1` redirect URLs enabled for production unless local auth testing is intentionally in scope for that session.

SMTP / sender configuration:

- Sender domain must be a verified `zambudget.com` domain.
- Preferred no-reply sender: `auth@zambudget.com` if that mailbox/domain identity is verified.
- Sender name: `Zam! Security`.
- Public support contact in templates: `support@zambudget.com`.
- Do not use `auth@budget-buddy.io`, `support@budget-buddy.io`, or `BudgetBuddy Security` after the Zam cutover.

## Email Template Requirements

Every Supabase Auth email template should be reviewed after a brand/domain cutover:

- Confirm email/signup template
- Magic link template
- Password reset template, if enabled
- Email change template, if enabled
- Reauthentication / OTP template, if enabled

Template copy requirements:

- Use `Zam!`, not `BudgetBuddy`.
- Use `Zam! Security`, not `BudgetBuddy Security`.
- Use `support@zambudget.com`, not any `budget-buddy.io` address.
- Keep the warning: Zam! will never ask for a Cloud Sync recovery key by email.
- Keep auth links generated by Supabase's template variables; do not hardcode confirmation tokens or auth URLs.

## App Code Guardrail

The login page and in-app password reset flow must not generate production Supabase auth emails with localhost callback links.

Current guardrail:

- `login.html` uses `getAuthRedirectUrl(...)`.
- `js/ui.js` uses `getAuthRedirectUrl(...)` for password reset.
- On `localhost` and `127.0.0.1`, auth redirects fall back to `https://app.zambudget.com`.

Do not replace these helpers with `new URL(..., window.location.href)` for Supabase auth emails unless local auth redirect testing is deliberately reintroduced and documented.

## Test Guardrail

Automated login tests must not hit the real Supabase Auth service.

- `tests/login.spec.js` must intercept `js/vendor/supabase.js`.
- Do not leave login tests intercepting the old CDN path.
- If this test stub fails, test addresses such as `new@example.com` can trigger real Supabase email attempts.

## Verification

After changing auth settings:

- Send one test signup or magic-link email to a real disposable inbox you control.
- Inspect the raw message headers/body.
- Confirm `From` uses a `zambudget.com` sender.
- Confirm subject/body say `Zam!`.
- Confirm support address is `support@zambudget.com`.
- Confirm the auth link redirects to `https://app.zambudget.com/...`.
- Confirm no `budget-buddy.io`, `BudgetBuddy`, `localhost`, or `127.0.0.1` remains.
- Confirm login and demo smoke tests still pass.
